# 阅读笔记

## 一句话总结

很多看似应该被禁止的“非法状态”，其实只是“不想要的状态”：系统应该能够表示它们、检测它们，并引导它们最终被解决。

## 关键概念

- **Illegal state / 非法状态**：系统任何时候都不应进入。
- **Unwanted state / 不想要的状态**：系统可以短暂进入，但不应长期停留。
- **Invariant / 不变式**：非法状态通常对应不变式被破坏。
- **Liveness / 活性**：不想要的状态常常要求“最终会离开”。

## 可延伸思考

- 在业务系统里，哪些约束应该做成“无法表示”？
- 哪些状态虽然危险，但应该允许临时存在？
- 对用户而言，“明确告知正在进入不想要状态”是否比直接禁止更好？

## 我的摘录

> An illegal state is a state we never want our system to be in. An unwanted state is a state we don't want to stay in.

## 讨论记录：文章在讲什么

这篇文章主要是在提醒我们：软件设计里要区分两种状态——**非法状态（illegal state）**和**不想要的状态（unwanted state）**。很多工程师以为应该通过类型、数据库约束或业务规则彻底禁止的状态，其实并不是“非法”，而只是“不想要”。

### 核心区别

- **非法状态**：系统在任何时候都不应该进入。一旦进入，就说明核心不变式被破坏了。例如飞机实际起飞时乘客数超过座位数、数据结构违反一致性约束等。
- **不想要的状态**：系统可以短暂进入，但不应该长期停留。例如日历里出现重叠会议、航班暂时超售、系统发生网络分区等。

一句话说：非法状态要防止发生；不想要的状态要允许表达，但不能放任不管。

### 为什么不能把所有“不舒服”的状态都禁止掉

文章反驳了一种常见工程直觉：既然“让非法状态不可表示”是好原则，那是不是所有看起来危险或麻烦的状态都应该变成不可表示？作者认为答案是否定的。

现实世界、外部输入和业务流程经常会把系统推入某些“不干净”的中间状态。如果系统完全不能表示这些状态，就无法检测、提醒和修复它们。过早禁止这些状态，可能会让模型看起来更干净，但也会让系统不再贴近真实业务。

### 例子：日历冲突

一个用户同一时间有两个重叠日程，看起来像是错误，但不一定是非法状态。用户可能：

- 暂时接受两个邀请，之后再决定去哪一个；
- 一个是线上会议，一个是提醒事项；
- 一个只是占位；
- 或者用户就是有意这样安排。

所以，日历冲突更像是不想要的状态。系统应该明确提示冲突，而不是一定让这种状态不可表示。

### 例子：航班超售

航班超售也是不想要的状态，而不一定是非法状态。航空公司可能有意允许超售，因为预计会有乘客不出现。

真正非法的是：飞机实际起飞时，机上乘客数超过座位数。

因此可以区分：

- 超售：不想要的状态，业务上可能允许；
- 超载飞行：非法状态，必须禁止。

### 例子：网络分区

网络分区也是典型的不想要状态。我们无法保证网络永远可靠，所以要求“网络分区永远不发生”通常不现实。更合理的要求是：系统可以进入分区状态，但最终必须恢复，并且不能因为分区导致数据损坏。

### 设计启发

这篇文章对业务系统建模的启发是：不要急着把所有异常、冲突、临时不一致都设计成不可表示。

更好的问题是：

- 这个状态是否真的违反系统核心不变式？
- 它是否只是业务流程中的临时中间状态？
- 用户或外部世界是否可能合理地制造这个状态？
- 系统应该禁止它，还是应该检测、展示、告警并引导解决？

如果是非法状态，就应该通过不变式彻底禁止；如果是不想要的状态，就应该让系统能够表示它、看见它，并最终离开它。

## 讨论记录：TLA+ 形式化理解

### 状态序列

TLA+ 关注的不是单个静态状态，而是系统随时间变化形成的一整条执行轨迹：

```text
s0 -> s1 -> s2 -> s3 -> ...
```

每个 `s` 都代表系统某一刻的完整状态。文章中讨论的“非法状态”和“不想要的状态”，本质上是在问：这条状态轨迹中允许出现什么、不允许出现什么，以及出现后是否必须最终离开。

### `[] ~Illegal`：非法状态永远不能发生

文章里的 `[]!Illegal` 可以按 TLA+ 更常见的写法理解为：

```tla
[] ~Illegal
```

含义是：在所有时间点，系统都不能处于 `Illegal` 状态。

- `[]` 表示 always，即“始终”。
- `~` 表示 not，即“非”。
- `Illegal` 表示当前状态是非法状态。

所以 `[] ~Illegal` 表达的是：坏事永远不能发生。只要某一刻进入了 `Illegal`，即使后面恢复了，也已经违反了性质。

这类性质是典型的 safety property（安全性属性），也常对应系统 invariant（不变式）。

例如：

```tla
[] ~(actualPassengers > seats)
```

表示飞机实际载客数永远不能超过座位数。

### `[] <> ~Partition`：不想要的状态可以出现，但必须最终离开

文章里的网络分区例子可以写成：

```tla
[] <> ~Partition
```

含义是：从任意时间点往后看，最终都会再次到达非分区状态。

- `<>` 表示 eventually，即“最终”。
- `~Partition` 表示当前不处于网络分区。
- `[] <> ~Partition` 表示任何时候开始观察，未来都必须最终恢复到非分区状态。

这和 `[] ~Partition` 不同。后者要求网络分区永远不发生，现实中通常太强；前者允许分区发生，但不允许系统永远停留在分区中。

这类性质是 liveness property（活性属性）：它不只是说“坏事不发生”，而是在说“好事最终会发生”。

### Safety vs Liveness

可以用一句话区分：

- Safety：坏事永远不会发生。
- Liveness：好事最终会发生。

对应到本文：

- 非法状态通常用 safety / invariant 表达：`[] ~Illegal`。
- 不想要的状态常常需要 liveness 表达：`[] <> ~Unwanted`。

### 例子：日历冲突

如果 `Conflict` 表示用户有重叠日程，强行禁止可以写成：

```tla
[] ~Conflict
```

但这可能过强，因为用户可能有意接受两个冲突事件。更合理的性质可能是：

```tla
[] (Conflict => ConflictVisibleToUser)
```

也就是：只要存在冲突，就必须清楚地展示给用户。

或者在某些业务里，可以要求：

```tla
[] <> ~Conflict
```

即冲突可以短暂存在，但最终应被解决。

### 例子：航班超售

可以区分两个谓词：

```tla
Overbooked == soldTickets > seats
Overflight == actualPassengers > seats
```

航空公司可能希望允许 `Overbooked`，因为超售是业务策略；但必须禁止 `Overflight`，因为实际超载飞行是非法状态。

因此真正关键的性质可能是：

```tla
[] ~Overflight
```

而不是：

```tla
[] ~Overbooked
```

“不超售”比“不超载飞行”更强，因为如果永不超售，通常可以推出不会超载飞行；但反过来不成立。更强的性质不一定更好，因为它可能过度限制真实业务流程。

### 总结

本文的形式化启发是：不要把所有“不舒服”的状态都建模成非法状态。

- 真正的非法状态应该通过不变式禁止：`[] ~Illegal`。
- 现实世界或业务流程需要暂时存在的状态，应被建模为不想要的状态：允许表示、检测、提示，并用类似 `[] <> ~Unwanted` 或 `[] (Unwanted => Visible)` 的性质约束它。
